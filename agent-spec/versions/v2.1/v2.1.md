# Sema4.ai Agent Package Specification v2.1

## Agent Package File Structure

An Agent Bundle is a directory structure that contains the following components:

```
my-agent[.zip]
├── actions
│   └── MyActions
│       └── geo-actions # folder with sources
│       └── cool-shared-action
│           └── <action package files>
│   └── Sema4.ai
│       └── search
│           └── 1.0.0.zip
├── agent-spec.yaml
├── knowledge
│   └── Knowledge1.pdf
├── semantic-data-models
│   └── customer-analytics.yaml
└── runbook.md
```

The formal specification of the `agent-spec.yaml` is available in [agent-package-specification-v2.1.json](agent-package-specification-v2.1.json).

Note: the version in the spec is still referenced as `v2`, not `v2.1` due to backward
compatibility issues (as currently we don't have a good control where old versions
of the cloud in client receive a `v2.1` -- they currently break hard if faced with
a version they don't recognize).

An example of an agent spec is available in [example/agent-spec.yaml](example/agent-spec.yaml).

## New Features in v2.1

### MCP (Model Context Protocol) Servers

The v2.1 specification introduces support for MCP servers, allowing agents to connect to external
services and tools through the Model Context Protocol.

It also allows for the conversation-guide to be defined in the agent specification.

#### MCP Server Configuration

MCP servers are configured in the `mcp-servers` section of the agent specification:

```yaml
mcp-servers:
  - name: file-system-server
    transport: stdio
    description: MCP server for file system operations
    command-line: ['uv', 'run', 'python', '-m', 'mcp_file_system']
    env:
      FILE_SYSTEM_ROOT: /data
      MY_OAUTH2_API_KEY:
        type: oauth2-secret
        description: Your OAuth2 API key for authentication
        provider: Microsoft
        scopes:
          - user.read
          - user.write
      MCP_API_KEY:
        type: secret
        description: Your API key for authentication
      CONTENT_TYPE:
        type: string
        description: Your content type for authentication
        value: application/json
    cwd: ./mcp-servers/file-system
    force-serial-tool-calls: false
  - name: database-server
    transport: streamable-http
    description: MCP server for database operations
    url: http://localhost:8080/mcp
    headers:
      Authorization:
        type: oauth2-secret
        description: Your OAuth2 API key for authentication
        provider: Microsoft
        scopes:
          - user.read
          - user.write
      X-API-Version:
        type: string
        description: API version header
        value: '1.0.0'
      Content-Type: application/json
      X-email-API-Key:
        type: secret
        description: API key to access e-mails
        value: <optional-dev-value>
    force-serial-tool-calls: true
```

#### Supported MCP Server Types

1. **stdio**: Local MCP servers that run as subprocesses

   - Requires: `command-line`, optional `env`, `cwd`
   - Example: File system operations, local tools (note that docker
     gateway/toolkit should currently be used here, but special
     support is expected in the future to create a reproducible
     environment locally and in the cloud).

2. **streamable-http**: HTTP-based MCP servers

   - Requires: `url`, optional `headers`

3. **sse**: Server-Sent Events MCP servers
   - Requires: `url`, optional `headers`

#### Configuration Options

- **name**: Unique identifier for the MCP server
- **transport**: Transport of MCP server connection
- **description**: Human-readable description and configuration instructions
- **url**: Endpoint URL (for streamable-http and sse)
- **headers**: HTTP headers required for connection (for streamable-http and sse)
- **command-line**: Command to execute (for stdio)
- **env**: Environment variables needed (for stdio)
- **cwd**: Working directory (defaults to agent-spec directory)
- **force-serial-tool-calls**: Whether to serialize tool calls (default: false)

### Conversation Guide

The conversation guide just points to a yaml file that contains the actual conversation guide.

```yaml
conversation-guide: conversation-guide.yaml
```

### Docker MCP Gateway configuration

The Docker MCP Gateway is a tool that allows you to run MCP servers in Docker containers.

Docker MCP Gateway is configured in the `docker-mcp-gateway` section of the agent specification:

```yaml
docker-mcp-gateway:
  catalog: ./custom-catalog.yml # Optional: custom catalog file path
  servers:
    my-server:
      tools: [tool1, tool2] # Optional: whitelist of tools
    another-server:
      tools: [] # Empty or not specified means all tools are allowed
```

#### Configuration Options

- **catalog**: Optional custom catalog file path. If not specified, the default catalog from Docker MCP Gateway is used
- **servers**: Configuration for MCP servers in the Docker MCP Gateway
  - **tools**: Optional whitelist of tool names for a server. If empty or not specified, all tools are allowed

This configuration allows you to:

- Use a custom catalog for Docker MCP Gateway
- Control which servers should be used and which tools are available for each server

### Semantic Data Models

The v2.1 specification introduces support for semantic data models, allowing agents to package and distribute data models alongside the agent.

Semantic data models are configured in the `semantic-data-models` section of the agent specification:

```yaml
semantic-data-models:
  - name: customer-analytics.yaml
  - name: sales-model.yaml
```

#### Semantic Data Model Files

- Semantic data models are stored in the `semantic-data-models/` folder within the agent package
- Each model is a **YAML file** following the [Snowflake Cortex Analyst Semantic Model specification](https://docs.snowflake.com/en/user-guide/snowflake-cortex/cortex-analyst/semantic-model-spec)
- Data connection bindings are NOT included in the package (they must be configured after import)
- Models are referenced by filename only (similar to knowledge files)

#### Configuration Options

- **name**: Filename of the semantic data model (relative to `semantic-data-models/` folder)

**Note:** Data connections referenced by semantic data models are environment-specific and must be configured after importing an agent package.
